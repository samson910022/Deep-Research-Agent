## 3. 一致性檢查

### ⚠️ 發現 1：F3 嚴重性「High」但修復優先級為「P1」

- **審計報告**：F3（動態 pip install → 供應鏈 RCE）標示為 🔒 **High**
- **修復計畫**：F3 對應 **P1**（本 sprint，非上線前）
- **問題**：供應鏈 RCE 是典型的 **P0** 等級漏洞。雖然修復計畫中合理論述了動態安裝與鎖版本是兩層防護（P1-1 移除動態安裝 + P3-1 鎖版本），但 **High + RCE 風險 應至少列為 P0 或 P0.5**。如果攻擊者在上線前就透過 MITM 注入惡意套件，後果與 F2 同等嚴重。
- **建議**：將 P1-1 提升至 P0 或新增 P0-6，或至少在審計報告中提供為何 High 但不需要 P0 的補充說明。

### ⚠️ 發現 2：F8 嚴重性「Medium」但實際風險接近 High

- **審計報告**：F8（未驗證使用者查詢 → Subagent prompt injection）標示為 🔒 **Medium**
- **理由檢查**：審計報告原文指出「subagent 有 terminal 權限」。如果使用者查詢可注入到 subagent，且 subagent 有 terminal access，這條攻擊鏈：`惡意查詢 → prompt injection → subagent 執行任意命令 → 主機淪陷` 與 F2 的 `惡意 DOI → curl → RCE` 路徑相似。**Medium 偏低，應為 High。**
- **建議**：將 F8 從 Medium 升級至 High，或至少在報告中補充降級理由（如「需要使用者主動介入，非自動觸發」）。

### ⚠️ 發現 3：F6 與 F5 合併為 P0-4 — 合理但可更清楚

- P0-4 同時涵蓋 F5（`User` 路徑）與 F6（`D:\` 無備援），但修復計畫中只寫了一個 Action 段落。兩者雖屬同類（硬編碼路徑），但影響不同（F5 使別台機器完全無法使用，F6 使單分割區機器 crash）。建議在 P0-4 下明確分為兩個子項目。

### ✅ 風險關聯圖正確性

```
F1 → 注入假論文      ✅ 合理
F2 → 惡意 DOI 觸發指令 ✅ 合理
F7 → 成本失控        ✅ 合理
F4 → 反幻覺保證失效   ✅ 合理
F8 → Prompt injection  ✅ 合理（但若升級為 High 則更符合圖中位置）
F3 → 供應鏈 RCE       ✅ 合理

F1 → F2 → 完整主機淪陷 ✅ 攻擊鏈描述準確
```

**圖中無事實錯誤。**

1. **F3 優先級調整**：將 P1-1（動態 pip install）提升至 P0，或在新段落補充為何 High RCE 風險可以等下一 sprint 才處理的商業理由
